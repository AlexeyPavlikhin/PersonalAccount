Цель: 
Необходимо реализовать новый раздел в приложении, предназначенный для гернерации типовых документов.

Задачи:
1) реализуй код приложения
2) подготовь срипты модификации структру БД для харнения конфигурации
3) подготовь пример шаблона договора в формте pdf


Требования:
Раздел должен быть реализован, как компонент и встроен в приложении по аналогии других компонентов, такоих как @components/pg_uc.js, @components/pg_sales.js, @components/pg_courses.js   
Раздел должен быть доступен в основном меню под названием "Генерация документов".
Права доступа к разделу должны выдаваться по аналогии с дргуими разделами приложения.
Генерация документов должна осуществляться на основе шаблонов. 
Шаблоны будут размещаться в облачном хранилище (будут доступны по ссылке). Примеры шаблонов документов находятся тут @requirements/docs_temlpates.
Ссылки на шаблоны документов нужно будет получать из конфигурации, хранящейся в БД. Эту конфигурацию будет наполнять человек в рамках другого процесса (вне скоупа текущей задачи).
Шаблон должен быть заполнен данными (реквезиты организации), полученными по api от TenderGuru
Для заполнения шаблона необходимо предоставить пользователю поле для ввода ИНН организации, а так же БИК. После этого приложение должно создать конечный документ на основе шаблона, заполнив его данными, полученными из api провайдера.
Шаблоны будут в формате pdf
Готовый документ должен быть подготовлен в формате pdf


Основаной перечень данных, необходимых для получения их API провайдера:
- наименование организации;
- ФИО подписанта;
- должность подписанта;
- основание полномочий;
- адрес;
- ОГРН;
- ИНН;
- КПП;
- расчётный счёт;
- банк;
- корреспондентский счёт;
- БИК;
- email;
- телефон.

Стек получени информации о контрагентах:
- DaData для поиска компании по ИНН;
- DaData Bank для автозаполнения банка по БИК;
- хранить р/с контрагента у себя в БД после первого ручного ввода.

Клиетнский путь:
1. Пользователь вводит ИНН.
2. Система тянет из DaData party:
- название,
- ИНН,
- КПП,
- ОГРН,
- адрес,
- руководителя,
- должность.
3. Пользователь вручную вводит расчётный счёт и при необходимости корректирует подписанта.
4. Пользователь вводит БИК.
5. Система тянет из DaData bank:
 - банк,
 - корсчёт,
 - SWIFT при необходимости.
