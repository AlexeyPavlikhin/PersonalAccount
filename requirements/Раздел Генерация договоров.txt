Цель: 
Необходимо реализовать новый раздел в приложении, предназначенный для гернерации типовых документов.

Задачи:
1) реализуй код приложения
2) подготовь срипты модификации структру БД для харнения конфигурации
3) подготовь пример шаблона договора в формте docx


Требования:
1) Раздел должен быть реализован, как компонент и встроен в приложении по аналогии других компонентов, такоих как @components/pg_uc.js, @components/pg_sales.js, @components/pg_courses.js   
2) Раздел должен быть доступен в основном меню под названием "Генерация документов".
3) Права доступа к разделу должны выдаваться по аналогии с дргуими разделами приложения.
4) Генерация документов должна осуществляться на основе шаблонов. 
5) Шаблоны будут размещаться в облачном хранилище (будут доступны по ссылке). Примеры шаблонов документов находятся тут @requirements/docs_temlpates.
6) Ссылки на шаблоны документов нужно будет получать из конфигурации, хранящейся в БД. Эту конфигурацию будет наполнять человек в рамках другого процесса (вне скоупа текущей задачи).
7) Шаблон должен быть заполнен данными (реквезиты организации), полученными по api от DaData.
8) Для заполнения шаблона необходимо предоставить пользователю поле для ввода ИНН организации, а так же БИК. После этого приложение должно создать конечный документ на основе шаблона, заполнив его данными, полученными из api провайдера.
Логика заполнения полей ИНН и БИК:
 - поле ИНН всегда заполняется пользователем
 - поле БИК заполняется:
    - либо значение подтягивается из БД (туда они может быть сохранено в привязке к ИНН ранее при предыдущих запусках формирования данного шаблона по данному ИНН)
    - либо заполняться пользоватлем (если в БД информация отсуствует)
 поля ИНН и БИК - обязательны к заполнению   
9) Шаблоны будут в формате docx
10) Готовый документ должен быть подготовлен в формате docx

11) Поля "расчётный счёт", "email" и "телефон" после заполнения (генерации документа) должны сохраняться в БД с привязывзкой к ИНН организации.
12) Дозаполнение формы данными полученными по ИНН, БИК и из локальной БД должно происходить по нажатии одной единственной кнопки "дозаполнить данные".
13) должна осуществляться валидация расчетного счета и корресспондентского счета в соответсвии с правилами валидации счетов.

14) Дата составления договора должна браться из предзполненного текущей датой поля (с возможностью редактирования).
15) В документе Месяц может встерчаться как в формате числа, так и прописью
16) В шаблоне документов должны быть поля, содержащие элементы даты: отдельно день, отдельно месяц (прописью), отдельно год (как в формате YYYY, так и в формате YY)
17) Данные, полученные из DaData должны склоняться и менять регистор. Например  
    - поле "Должность подписанта": из DaData получено "ГЕНЕРАЛЬНЫЙ ДИРЕКТОР", а в документ нужно внести "генерального директора",
    - поле "ФИО подписанта": из DaDatr получено "Шевченко Сергей Иванович", а в документ нужно ввести "Шевченко Сергея Ивановича"
18) В документе имя и отчество подписанта должно быть указано как полностью (например в первой части договора), так и инициалами (например в разделе подписей)
19) В документе должны быть поля, зависящие от пола лица (контрагента), если это физическое лицо. Значени: именуемый - для мужского пола, именуемая - для женского. 

20) должна быть реализована возможноть формирования документа содержащего название организации-контрагента (полученного из DaData), как с коротким названием ОПФ, так и с полным ОПФ.  

21) все данные, которые кэшируются нужно кэшировать без привязки к id шаблону. 
22) перечень кешируемых/восставаливаемых из кэша полей нужно хранить (настраивать) в таблице document_templates в отдельном поле.
23) название поля, содержащего ключевую информациюдля привязки кэшируемых данных так же долно храниться в отдельно моле таблицы document_templates.
24) сами закешированные данные должны хрниться в отдельной таблице, которая должна иметь в своей структуре поле содержащее значение ключевой информации (например значение ИНН), поле, содержащее имя поля ключевой информации (например inn) и поле, содержащее сами закэшированные данные в формате  ключ - значение. Примем ключи могут быть не уникальными. В этом случае в поля, куда будет производиться востановление закэшированных данных должна быть предоставлена возможность выбора одного значения из нескольких (наприер выпадающий список).  
25) кэширование должн происходить при генерации документа по шаблону.
26) Логику восставнолвения данных из кэша: если поле уж есодержит информацию, то данные из кэша в это поле не восстанваливать 
27) если появлется множестванный выбор в полях, где произошло восставновлени из кэша, то поля с выпадапющим списком должны иметь желтый фон (именно поле св выпапдющим списком, а не оба связанных поля (включая поле "конечного выбора"))
28) В случае нескольких восставновленных из кэша начений, необходимо по-умолчанию использовать последнее по дате


 Требования к UI:
 1) Пользовательская форма должна быть выполнена в корпративном стиле
 2) Пользователю должен быть предоставлен список шаблонов с возможностью фильтрации
 4) После выбора шаблона, пользователю должно быть предложено заполнить доплнительные данные: указать ИНН, БИК или что-то другое. Перечень дополнитеоный информации должен быть настраиваемый для каждого шаблона.
 
 5) На форме должны присутствовать кнопки "Редактироаать" и "Сформировать DOCX". 
 6) На форме должна присутствовать область с предварительным просмотром "проекта документа", полученного из шаблона путём заполнения полей.
 7) При нажатии на кнопку "Редактировать" должно появиться модальная панель содержащая поля для заполнения (конфигурация из таблицы document_template_fields).
 8) На панели редактирования должны быть кнопки "Дозаполнить данные" с логикой, описанной в разделе п. 12 текущих требований, "Сохранить", принажатии  нак которую должно происходить заполнение шаблона и закрытие панели и "Отменить", при нажатии на которую происходит закрытыие панели без сохранения параметром м заполнения шаблона.
 9) если при попытке дозаполнить данные  обязательные поля незаполнены (или не прошли валидацию), то помимо информационного сообщения необходимо подсвечивать незаполненные/проблемные поля красной рамкой. В случае с использования полей с выбором из множества восстановленных из кэша данных, выделять красной рамкой нужно именно поля "конечного выбора"
 
 10) При запуске формы пользователю должен быть предоставлен список оформленных ранее договоров и привязанных к ним оформленных спецификаций и счетов. 
 11) Таблица оформленых договоров должна содержать следующие поля: 
 - Номер договора
 - Дата составления договора
 - Коротокое описание предмета договора
 - Контрагент
 - Номер спецификации, 
 - Дата составления сецификации,
 - Номер счета,
 - Дата выставления счета
 12) Приложение должно предоставлять возможность как "ручного редактирования" таблицы оформленных договоров, так и автоматизированого заполнения таблицы  в результате создания соответствующих докуметов (договоро, счетов, актов)
 
 13) При переходе в раздел "Генерация документов" пользователю должено отобразиться блок "Оформленные договоры" на всё рабочее пространство. Т.е. фильтр "Шаблоы локументов" и блок, содержащий просмотровщик шаблона, отображаться не должен.
 14) В блоке "Оформленные договоры" должна быть кнопка "Шаблоны документов", при нажатии на которую должен должны появиться блоки "Шаблоы локументов" и блок с посмотровщиком. При этом блок "Оформленные договоры" виден быть не должен.
 15) После завершенияч работы с шаблонами (нажатии на кновку "Сформировать DOCX"), блок "Шаблоы локументов" и блок с посмотровщиком долны скрыться и должен снова появиться блок "Оформленные договоры". Так же в блоке с просмотровщиком должна быть кнопка "Отменить", при нажатии на которую так же блок "Шаблоы локументов" и блок с посмотровщиком долны скрыться и должен снова появиться блок "Оформленные договоры". 

Требования к формированию, обработке и учету отдельных полей: 
1) Введённые/сгенерированные данные для полей "Номер договора", "Дата составления договора", "коротокое описание предмета договора", Контрагент ("Наименование (краткое ОПФ)", или "Наименование ИП", или "ФИО") должны сохранияться в момент нажатия кнопки "Сформировать DOCX" в отдельной таблице.
2) Введённые/сгенерированные данные для полей "Номер спецификации", "Дата составления сецификации", "Номер счета", "Дата выставления счета" должны сохранияться в момент нажатия кнопки "Сформировать DOCX" в отдельной таблице в привязке к номеру договора.
3) Поле "Номер договора" должно иметь следующий формат: XX-MM/YY, где XX - сквозная нумерация всех заключаемых договоров и заполняться автоматически с инкриментом +1 к последнему номеру, сохраненному в ссответсвующей таблице, MM - месяц даты заключения договора (поле должно заполняться автоматически), YY - год даты заключения договора (поле должно заполняться автоматически). 
4) Поле "Номер спецификации" должно иметь последовательную нумерацию в рамках одного договора и заполняться автоматически с инкриментом +1 к последнему номеру, сохраненному в соответствующей таблице.
5) Поле "Номер счета" должно иметь сковзную нумерацию для всех спецификация и заполняться автоматически с инкриментом +1 к последнему номеру, сохраненному в соответствующей таблице.
6) Шаблон может содержать табоицы, поэтому приложение должно уметь осуществлять ввод табличных данных и заполнение шаблона табличными данными.

Основаной перечень данных, необходимых для получения их API провайдера:
- наименование организации;
- ФИО подписанта;
- должность подписанта;
- основание полномочий;
- адрес;
- ОГРН;
- ИНН;
- КПП;
- расчётный счёт;
- банк;
- корреспондентский счёт;
- БИК;
- email;
- телефон.

Стек получени информации о контрагентах:
- DaData для поиска компании по ИНН;
- DaData Bank для автозаполнения банка по БИК;
- хранить р/с контрагента у себя в БД после первого ручного ввода.

Клиетнский путь:
1. Пользователь вводит ИНН.
2. Система тянет из DaData party:
- название,
- ИНН,
- КПП,
- ОГРН,
- адрес,
- руководителя,
- должность.
3. Пользователь вручную вводит расчётный счёт и при необходимости корректирует подписанта.
4. Пользователь вводит БИК.
5. Система тянет из DaData bank:
 - банк,
 - корсчёт,
 - SWIFT при необходимости. 

 Документация на api dadata:
 API "банк по БИК": https://dadata.ru/api/find-bank/
 API "организация по ИНН или ОГРН": https://dadata.ru/api/find-party/
