# README

## Назначение

Веб-приложение личного кабинета (ЛК) для домена вокруг `msll-ip.ru` / `msll-dev`.

Ключевые задачи:
- работа с пользователями и клиентскими профилями;
- ведение продаж, заказов и статусов;
- управление курсами и доступами;
- интеграция с внешним сайтом оплаты (входящие уведомления о платеже);
- отправка уведомлений по e-mail через `PHPMailer`.

## Технологический стек

| Слой | Технологии |
|---|---|
| Сервер | `PHP`, сессии `session_start()` |
| База данных | `MySQL` через `PDO` (`config.php`, `dev_config/config.php`) |
| Фронтенд (ЛК) | `Vue 3` (локальные `js/vue.global.js`), `vue-router` (hash history), `axios`, `vue-spinner` |
| Вёрстка | статические CSS (`css/styles.css`, `jost.css`) |
| Почта | `PHPMailer` из `libs/PHPMailer-master/` |
| Локальная разработка | `WAMP` (`c:\wamp64\www\1`) |

Примечание: отдельной сборки фронта (`Vite/Webpack`) для основного приложения в корне нет. Vue подключается готовыми бандлами из `js/`, компоненты вынесены в ES-модули `.js` в `components/`.

## Структура проекта (рабочая часть)

- `index.php` — точка входа в SPA ЛК: проверка сессии и группы пользователя (`client` / `operator`), монтирование Vue, маршруты `/`, `/uc`, `/sales`, `/courses`.
- `login.php` — авторизация (`password_verify`), запись событий в таблицу `audit`, разветвление сообщений для `mobile/desktop`.
- `lk.php`, `order_details.php`, `order_editor.php` и др. — серверные PHP-страницы с HTML-рендерингом (часть функционала заказов/админки работает параллельно SPA).
- `queries/*.php` — HTTP-эндпоинты под `axios` (GET/POST): CRUD клиента, списки, справочники, обновление пользователей и пр. (около 50 файлов).
- `api/` — внешние API-эндпоинты (`payment_event.php`, `send_notification.php`, `sync_emails.php` и др.): разбор запроса (`parse_str` от `php://input`), проверка API-ключа через `ALLOWED_APIKEY`, работа с БД и почтой.
- `config.php`, `prod_config/`, `dev_config/` — параметры БД, SMTP, разрешённых хостов и API-ключа.
- `components/` — Vue-компоненты как `.js` модули (формы редактирования клиента, навигация, страницы `pg_*`).
- `trash/` — черновики, старые копии и примеры (обычно не часть продуктового контура).

## Архитектура взаимодействия

1. Пользователь открывает `index.php`.
2. PHP проверяет сессию и права доступа.
3. Отдаётся HTML-оболочка и клиентские скрипты.
4. `Vue Router` переключает представления.
5. Компоненты запрашивают данные через `./queries/*.php` (`axios`).

Интеграция оплаты:
- внешний сайт (корзина/оплата) отправляет `POST` в `api/payment_event.php`;
- по продукту ищется курс в справочниках;
- создаётся/обновляется пользователь;
- выдаётся доступ и выполняются сопутствующие обновления.

Рассылки:
- `api/send_notification.php` используется для e-mail уведомлений/напоминаний (например, по срокам доступа) и ведения аудита вызовов.

## Данные и предметная область

Основные сущности (по фактической схеме в `database/tables.sql` и по коду):
- пользователи ЛК: `users` (в т.ч. `user_group`: `client` / `operator` и др.);
- клиентский профиль: `clients` и связанные `clients_email`, `clients_phone`, `clients_telegram`;
- каталог продуктов и продажи: `products`, `sales`, справочник статусов `sales_status`;
- курсы и контент: `spr_courses_name`, `spr_courses_contents`, `spr_courses_item_types`, `courses_data`, выдача доступа `users_premited_courses`;
- связь «имя продаваемого объекта → курс»: `spr_link_sale_cource` (имя таблицы с опечаткой в схеме);
- права/разрешения: `spr_permitions`, назначения `users_permitions`;
- заказы: `orders`;
- журнал: `audit`, служебный лог загрузок: `loadlog`.

## Структура базы данных

### Где лежит схема

- `database/tables.sql` — дамп структуры таблиц (phpMyAdmin), база в заголовке: **`msll_lk_db`**, сервер в дампе: **MySQL 8.0**, кодировка соединения: **utf8mb4** (`utf8mb4_0900_ai_ci` для большинства полей; таблица `users` — `utf8mb3` / `utf8mb3_unicode_ci`).
- `database/procedures.sql` — фрагмент дампа **хранимых процедур**; фактически файл **неполный / повреждённый**: после объявлений процедур идут обрывки `DELIMITER`/`BEGIN` без корректных тел и завершения. Имена процедур, которые задумывались в дампе: `concat_clients_property`, `insert_new_string`, `united_client`. Для точного описания логики нужен **полный актуальный дамп** процедур с сервера или восстановление из бэкапа.

### Таблицы и назначение (кратко)

| Таблица | Назначение |
|---|---|
| `users` | Учётные записи ЛК: логин, e-mail, хэш пароля, группа (`user_group`). |
| `clients` | Карточка клиента: ФИО, работа, комментарий, опциональная связь `user_id`. |
| `clients_email`, `clients_phone`, `clients_telegram` | Контакты клиента (1:N), внешний ключ на `clients.client_id`; уникальность e-mail/телефона/telegram в рамках своих таблиц. |
| `products` | Справочник продуктов (уникальность имени по индексу). |
| `sales_status` | Справочник статусов продажи. |
| `sales` | Продажа: клиент, продукт, дата, комментарий, ссылка на статус. |
| `orders` | Заказ: внешний `order_id`, клиентская привязка, статус, описание, время создания строки, оператор. |
| `spr_courses_name` | Курс: имя, длительность в днях, опциональная дата старта. |
| `spr_courses_contents` | Элементы содержания курса (название, сортировка), FK на курс. |
| `spr_courses_item_types` | Тип элемента контента (текст/ссылка и т.д. — по смыслу справочника). |
| `courses_data` | Данные по элементу содержания: два текстовых поля полезной нагрузки, тип, сортировка; FK на `spr_courses_contents` и `spr_courses_item_types`. |
| `spr_link_sale_cource` | Сопоставление строки продаваемого объекта (`sale_object_name`) с `course_id`. |
| `spr_permitions` | Справочник разрешений (с группировкой `permition_group`; в именах полей — опечатка `permition`). |
| `users_permitions` | Связь пользователь ↔ разрешение, срок `deadline`. |
| `users_premited_courses` | Доступ пользователя к курсу до даты `available_until` (в имени таблицы опечатка `premited`). |
| `audit` | Журнал: время, логин, тип операции, текстовые детали события. |
| `loadlog` | Простой лог (`MyISAM`): дата и текст записи. |

### Связи и индексы (важное)

- Целостность по продажам и курсам задана **внешними ключами** (`sales` → `clients`, `products`, `sales_status`; `courses_data` → справочники; `users_permitions` → `users` и `spr_permitions`; `users_premited_courses` → `users` и `spr_courses_name` и др.) — см. конец `database/tables.sql`.
- В схеме **`orders.client_id` внешне ссылается на `users.id`**, а не на `clients.client_id` — это стоит учитывать при чтении кода и миграциях (семантика поля может отличаться от названия «client»).

## Безопасность и техдолг

Позитивные моменты:
- в `login.php` используется `password_verify`;
- в ряде запросов применяются подготовленные выражения.

Риски, требующие доработки:
- в `api/payment_event.php` и отдельных участках встречается SQL через конкатенацию строк из входных данных (риск SQL-инъекций);
- в `login.php` функция `write_log` формирует `INSERT` через конкатенацию (аналогичный риск);
- секреты (пароли БД, SMTP, API key) хранятся в репозитории в открытом виде.

Рекомендуемые шаги:
- перевести SQL на параметризованные запросы во всех эндпоинтах;
- вынести секреты в переменные окружения или внешнее хранилище;
- выполнить ротацию текущих ключей/паролей;
- добавить регламент аудита и проверки конфигов перед деплоем.
